# アプリ開発ワークフロー

**最終更新日**: 2025年12月5日

このドキュメントは、Medical Prompt Hubやpediatric-exam-appなどのサービスを開発からリリースまで導くためのワークフローを定義しています。各段階で必要なチームとその役割、チェックポイントを明確にすることで、効率的かつ品質の高い開発を実現します。

## フェーズ1: 企画・ビジョン策定

**期間**: 1-2週間  
**主要チーム**: プロダクト戦略チーム  
**目的**: サービスの核となるビジョンと方向性を明確にする

### 1.1 担当チームと役割

#### プロダクト戦略チーム（主導）
- **Steve Jobs**: 製品の「魂」と「フォーカス」を決定
  - サービスの本質的な価値提案を定義
  - 不要な機能を排除し、極限まで単純化
  - 感情に訴える体験の方向性を提示
  
- **Peter Thiel**: 独占戦略の検討
  - 「賛成する人がほとんどいない、大切な真実は何か？」を問う
  - 小さなニッチ市場での独占可能性を検討
  
- **Clayton Christensen**: 顧客の「ジョブ」を特定
  - 顧客がこのサービスを「雇って」何を片付けようとしているか
  - 機能ではなく、顧客の状況と解決したい課題に焦点

- **Paul Graham**: 初期戦略の実践性を確認
  - 「人が欲しがるものを作れ」の観点から検証
  - スケールしないことをする（初期段階）の重要性を確認

### 1.2 成果物
- [ ] ビジョン文書
- [ ] コアバリュープロポジション
- [ ] 初期ターゲットユーザー定義
- [ ] 成功指標（Success Metrics）の定義

### 1.3 チェックポイント
**CP1: ビジョン承認会議**
- **参加チーム**: プロダクト戦略チーム、メディカルAI・倫理監修チーム（医療サービスの場合）
- **判断基準**: 
  - ビジョンが明確で一貫しているか
  - 医療サービスの場合、倫理的リスクが適切に評価されているか
- **承認条件**: 全参加チームの合意

  - 「競争は負け犬がすることだ」の観点から、競合を回避する方法を検討
  - 独占可能な市場領域の特定

#### データ分析・BIチーム
- **DJ Patil**: 市場データの分析
  - 市場規模と成長性の分析
  - データドリブンな市場理解
  
- **Hilary Mason**: 競合サービスの技術分析
  - 競合の技術スタックとアーキテクチャの分析
  - 技術的優位性の可能性を検討

### 2.2 成果物

- [ ] 競合分析レポート
- [ ] 差別化ポイントの明確化
- [ ] ターゲット市場の再定義

### 2.3 チェックポイント
**CP2: 市場戦略承認会議**
- **参加チーム**: プロダクト戦略チーム、データ分析・BIチーム
- **判断基準**: 
  - 市場機会が明確か
  - 競合優位性が確立できるか
- **承認条件**: 市場機会の確認と戦略の承認

## フェーズ3: プロダクト戦略・PMF

**期間**: 2-3週間  
**主要チーム**: プロダクト戦略チーム、メディカルAI・倫理監修チーム（医療サービスの場合）  
**目的**: プロダクト・マーケット・フィットを追求し、具体的な機能要件を定義する

### 3.1 担当チームと役割

#### プロダクト戦略チーム（主導）
- **Steve Jobs**: 機能の優先順位付けと削減
  - 不要な機能への「No」を明確に
  - 極限までの単純化
  
- **Paul Graham**: PMF追求の実践
  - 初期ユーザーとの直接対話の計画
  - 熱狂的な少数のファンを作る戦略

- **Clayton Christensen**: ジョブ理論の適用
  - 各機能が顧客の「ジョブ」をどう解決するか
  - 機能要件の優先順位付け

#### メディカルAI・倫理監修チーム（医療サービスの場合）
- **Eric Topol**: AIと医師の協働モデル
  - AI活用の方向性が医療を人間的にするか
  - 医師が患者と向き合う時間を増やす設計か
  
- **Atul Gawande**: 安全性の確保
  - エラー防止の仕組み
  - チェックリストの必要性

- **Gordon Guyatt**: 医学的信頼性
  - エビデンスベースの要件定義
  - ハルシネーション対策の要件

### 3.2 成果物
- [ ] プロダクト要件定義書（PRD）
- [ ] 機能優先順位リスト
- [ ] MVP（Minimum Viable Product）の定義
- [ ] ユーザーストーリー
- [ ] 医療サービスの場合：安全性要件、エビデンス要件

### 3.3 チェックポイント
**CP3: プロダクト要件承認会議**
- **参加チーム**: プロダクト戦略チーム、メディカルAI・倫理監修チーム、法務・知財チーム
- **判断基準**: 
  - 機能要件が明確で実現可能か
  - 医療サービスの場合、安全性と倫理的要件が満たされているか
  - 法的リスクが適切に評価されているか
- **承認条件**: 全参加チームの合意

  - リスク管理
  - 進捗管理

#### 品質保証・テストチーム
- **Kent Beck**: TDD推進
  - テスト駆動開発の指導
  - テスト戦略の確認
  
- **Martin Fowler**: リファクタリング支援
  - コード品質の維持
  - テストピラミッドの確認

#### コンテンツ・編集チーム（必要に応じて）
- **Andrej Karpathy**: プロンプトエンジニアリング
- **Strunk & White**: コンテンツの言語表現
- **NEJM Editor**: 医学的コンテンツの査読

### 6.2 開発プロセス

#### スプリントサイクル（2週間）
1. **スプリント計画**（1日目）
   - プロダクトバックログからスプリントバックログを作成
   - タスクの見積もりと割り当て

2. **開発**（2-9日目）
   - デイリースクラム（毎日15分）
   - TDDによる実装
   - コードレビュー

3. **スプリントレビュー**（10日目）
   - 完成した機能のデモ
   - ステークホルダーからのフィードバック

4. **スプリントレトロスペクティブ**（10日目）
   - 改善点の議論
   - 次のスプリントへの改善施策

3. **E2Eテスト**
   - 主要ユーザーフローのテスト
   - ブラウザ自動化テスト

4. **パフォーマンステスト**
   - ロードテスト
   - ストレステスト
   - Core Web Vitals測定

5. **セキュリティテスト**
   - 脆弱性スキャン
   - OWASP Top 10チェック

6. **アクセシビリティテスト**
   - WCAG AA準拠確認
   - スクリーンリーダーテスト

7. **探索的テスト**
   - 手動テスト
   - エッジケースの探索

## 注意事項

1. **フェーズの並行実行**: 一部のフェーズは並行して進めることができますが、依存関係を考慮してください。

2. **チェックポイントの厳格性**: 特にCP9（セキュリティ・コンプライアンス承認会議）は、リリース可否の最終判断となるため、全参加チームの承認が必要です。

3. **医療サービスの場合**: メディカルAI・倫理監修チームと医療規制・コンプライアンスチームの関与が必須となります。

4. **柔軟性**: このワークフローはガイドラインです。プロジェクトの特性に応じて調整してください。

5. **継続的改善**: リリース後も継続的に改善を繰り返し、PMFを追求し続けます。
